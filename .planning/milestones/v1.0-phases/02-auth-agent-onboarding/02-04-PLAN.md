Grant Bernard admin access and enforce admin-only route protection (AUTH-04). <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| /admin/* request -> middleware | Privilege boundary; only tokens carrying admin:true may pass |
| Node script -> Firebase Auth | Service account key grants claim-setting power; must never be committed or deployed |
| firebase-admin -> Workers runtime | Hard boundary: firebase-admin must NEVER be imported by app/Workers code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-17 | Elevation of Privilege | non-admin reaching /admin | mitigate | Middleware (Plan 02-01) reads admin claim from the signed JWT and returns 403; claim is tamper-evident |
| T-02-18 | Tampering | forging an admin claim client-side | mitigate | Custom claims live in the Firebase-signed ID token verified via Web Crypto; cannot be forged without Firebase's signing key |
| T-02-19 | Information Disclosure | service account key leakage | mitigate | serviceAccountKey.json gitignored; never deployed; used only by the local Node script |
| T-02-20 | Elevation of Privilege | firebase-admin reaching Workers and crashing runtime / breaking trust model | mitigate | admin-claim.test.ts asserts no src/ file imports firebase-admin; it is a devDependency confined to scripts/ |
| T-02-SC | Tampering | npm install (firebase-admin, tsx) | accept | firebase-admin is the official Google Admin SDK (devDependency, audited in RESEARCH); tsx is a widely-used TS runner — no [SUS]/[SLOP] requiring a blocking checkpoint |
</threat_model>
