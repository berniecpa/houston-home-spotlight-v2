<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form -> Firebase Auth | User credentials handled by Firebase client SDK (TLS); no plaintext password ever touches our Worker |
| Browser -> /api/auth/session | ID token POSTed for cookie exchange (verified server-side in Plan 02-01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Spoofing | login as unverified user | mitigate | Session route (Plan 02-01) rejects email_verified=false; LoginForm surfaces the rejection rather than granting access |
| T-02-08 | Elevation of Privilege | weak-password brute force | mitigate | Client enforces password >= 8 chars; Firebase Auth applies server-side rate limiting (RESEARCH Security Domain) |
| T-02-09 | Information Disclosure | verbose auth errors leaking account existence | accept | UI-SPEC copy intentionally distinguishes user-not-found vs wrong-password for UX; acceptable for a low-value agent marketplace, mitigated by Firebase rate limiting |
| T-02-10 | Tampering | XSS via reflected error/redirect param | mitigate | React escapes rendered error copy; redirect param used only for client router.push to same-origin paths, never rendered as HTML |
| T-02-11 | Repudiation | password reset abuse (mailbombing) | accept | sendPasswordResetEmail is Firebase-managed with built-in throttling; no custom token flow to abuse |
</threat_model>
