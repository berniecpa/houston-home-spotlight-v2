is NOT NULL, so a general inquiry cannot be inserted into leads).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public buyer → /api/leads | Untrusted inquiry input; listing/agent must be resolved server-side, not trusted from body |
| agent browser → /dashboard/leads | Authenticated read; an agent may see only their own leads |
| worker → Resend/Perfex | Outbound side effects; failures must not leak or block |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Spoofing | POST /api/leads | mitigate | Resolve listing_id + agent_id from listingSlug via D1 — never trust a body-supplied listing/agent id |
| T-04-12 | Information Disclosure | /dashboard/leads | mitigate | SELECT ... WHERE agent_id = session uid; no cross-agent lead visibility (LEAD-05) |
| T-04-13 | Information Disclosure | sendLeadEmail | mitigate | RESEND_API_KEY read from env only; never logged, never returned to client |
| T-04-14 | Denial of Service | best-effort side effects | mitigate | Promise.allSettled with per-branch catch+log; D1 insert is source of truth; success not coupled to email/CRM (LEAD-04) |
| T-04-15 | Tampering | leads INSERT + inbox query | mitigate | Parameterized prepare().bind() throughout — no string concatenation |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages this phase — Resend via raw fetch, no SDK |
</threat_model>
