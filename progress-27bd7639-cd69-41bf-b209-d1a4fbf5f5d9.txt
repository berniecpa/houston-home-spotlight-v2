## Codebase Patterns
- Project uses standard Next.js 14 App Router structure with src/ directory
- TypeScript strict mode enabled with @/* path alias to ./src/*
- Static export configured with output: 'export' and distDir: 'dist'
- Testing uses Node.js built-in test runner: node --test
- Images marked as unoptimized for static export compatibility
- Cloudflare Pages deployment via wrangler.toml with bucket = "dist"
- Types are defined in src/types/index.ts with JSDoc comments for documentation
- Test files follow the pattern src/tests/*.test.ts
- Field names in types should match external API expectations (e.g., Perfex CRM uses 'firstname', 'lastname', 'phonenumber')
- Data files stored in src/data/listings/ as individual JSON files
- Listing data utility functions in src/lib/data.ts with caching support
- Placeholder images via picsum.photos with seeded URLs for consistency
- Components use 'use client' directive for client-side interactivity
- Next.js Link component imported from 'next/link' for navigation
- Mobile-first responsive design with Tailwind classes (md: prefix for desktop)
- Semantic HTML elements (header, nav, footer, main) for accessibility
- ARIA labels and attributes for accessibility (aria-label, aria-expanded)
- Hero sections use gradient backgrounds with container-custom for consistent width
- SVG wave dividers create visual separation between sections
- Home pages fetch data at build time using async page components
- Responsive grid for listings: grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6 sm:gap-8
- CTAs use accent color (amber) for primary actions, outline style for secondary
- Filter components use fieldset/legend for semantic HTML grouping
- Input fields use inputMode="numeric" for mobile keyboards
- Button groups use role="radiogroup" and aria-checked for accessible selection
- Expandable sections on mobile use aria-expanded and aria-controls
- API routes are in src/app/api/* with route.ts files (Next.js App Router convention)
- API routes use NextRequest/NextResponse from 'next/server'
- Environment variables for external APIs use process.env.* pattern
- API error responses return appropriate HTTP status codes (400 for validation, 500 for server errors)
- API routes validate input before calling external services
- External API calls use fetch with proper headers and JSON body
